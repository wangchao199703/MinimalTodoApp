## 各项详细说明

### 1. 常驻桌面 + 托盘
- 标题栏右侧只有「最小化」和「关闭」两个键，**关闭键不退出程序**，而是 `Hide()` 到系统托盘。
- 程序真正退出的唯一入口：**托盘图标右键 → 退出**（`ExitMenuItem_Click` 中置 `_allowClose=true` 后 `Shutdown`）。
- 托盘双击 / 右键「显示主界面」可恢复窗口。
- 相关文件：`Views/MainWindow.xaml.cs`（`OnClosing`、`HideToTray_Click`、`ExitMenuItem_Click`）。

### 2. 分组栏宽度调整 / 隐藏 / 详情
- **调宽**：主体 Grid 左列 + `GridSplitter`，拖动改变 `SidebarColumn.Width`，关窗时写回 `AppData.SidebarWidth` 持久化。
- **隐藏/展开**：标题栏「☰」按钮触发 `ToggleSidebarCommand`，代码联动把左列宽度设为 0 或恢复持久化宽度；`SidebarCollapsed` 也持久化。
- **分组详情**：每个分组项右侧「ⓘ」`ToggleButton` 控制一个 `Popup`，显示分组名、颜色、未完成数量，并提供「删除该分组」。再次点击或点击别处即收起。

### 3. 新任务日期按需弹出
- 底部添加栏左侧「📅」按钮 → `ToggleNewTaskDateCommand` 切换 `IsNewTaskDatePickerOpen`。
- 仅当展开时 `DatePicker` 才可见；收起时自动清空 `NewTaskDueDate`，保证“平时无日期框”。

### 4. 新建分组按需弹出
- 默认显示「＋ 新建分组」按钮（`BeginAddGroupCommand` → `IsAddingGroup=true`）。
- 展开后显示输入框 + 「✓ 确定」「✕ 取消」；回车确定、Esc 取消。
- 命令：`ConfirmAddGroupCommand` / `CancelAddGroupCommand`。

### 5. 添加任务移到底部
- 右侧区域改为三行：标题/排序（顶）、任务列表（中，占满）、添加栏（底）。

### 6. 倒计时 + 右键操作
- `TodoItem` 新增计算属性：`DueState`（枚举）、`DueCountdownText`（“今天到期 / 明天到期 / 剩 N 天 / 已逾期 N 天 / 已完成”）、`HasDueDate`。
- 倒计时颜色由 `DueStateToBrushConverter` 按状态映射主题色（逾期红、临近橙、正常次要色）。
- `DispatcherTimer` 每分钟刷新一次，跨零点也能自动更新文案。
- 任务行**不再有日期选择框**；右键菜单提供：设置/修改截止日期…（弹 `DueDateDialog`）、清除截止日期、标记完成、删除任务。
- ContextMenu 因不在可视树，命令通过 `PlacementTarget.Tag`（Tag 绑定到窗口 DataContext）访问 ViewModel。

### 7. 自定义窗口样式
- `WindowStyle="None"` + `System.Windows.Shell.WindowChrome`：去掉系统标题栏，保留拖动、缩放、贴边、任务栏行为。
- 自绘 40px 标题栏（`TitleBarBg` 主题色），与窗体一致；标题栏按钮用 `Segoe MDL2 Assets` 图标字体。
- 按钮加 `WindowChrome.IsHitTestVisibleInChrome=True` 以便在标题区可点击。

### 8. 多套主题
- `ThemeManager` 泛化为支持任意多套主题（`ThemeManager.All`）。
- 主题文件均在 `Themes/`，键名完全一致，新增统一键 `TitleBarBg`、`WarningText`、`SuccessText`。
- 6 套：`Light` 明亮、`Dark` 暗黑、`Nord` 极地、`Ocean` 海洋、`Forest` 森林、`Rose` 玫瑰。
- 侧边栏底部「🎨」下拉切换，选择即时生效并持久化（`AppData.Theme`）。

### 9. 打包
- 命令（框架依赖，单文件，体积最小，目标机需 .NET 8 桌面运行时）：
  ```powershell
  dotnet publish MinimalTodoApp\MinimalTodoApp.csproj -c Release -r win-x64 `
    --self-contained false -p:PublishSingleFile=true -p:DebugType=none
  ```
- 产物：`bin\Release\net8.0-windows\win-x64\publish\MinimalTodoApp.exe`，**约 1.68 MB**，单文件可直接拷贝运行。
- 若目标机没有 .NET 8 运行时，改用自包含（体积约 70–90MB）：
  ```powershell
  dotnet publish MinimalTodoApp\MinimalTodoApp.csproj -c Release -r win-x64 `
    --self-contained true -p:PublishSingleFile=true -p:EnableCompressionInSingleFile=true
  ```

- 模型新增 `Priority`（None/Low/Medium/High）。

- 倒计时定时器从每分钟改为**每 30 秒**刷新。

### 6. 右键编辑（修复 + 增强）
- **修复**：旧版右键「设置截止日期」依赖的菜单项解析在 ContextMenu 场景下不稳定；改为优先用 `MenuItem.DataContext`（ContextMenu 继承 PlacementTarget 的 DataContext）直接拿到任务，并保留 PlacementTarget 兜底。

- 右键菜单：编辑(优先级/截止时间)… / 清除截止日期 / 标记完成 / 删除任务。
- ViewModel 新增 `ApplyTaskEdits(item, due, priority)` 统一回填。

## 总览
| # | 需求 | 状态 | 关键实现 |
|---|------|------|----------|
| 1 | 侧边栏展开时，图标与图标对齐、所有文字左边也对齐 | ✅ | 每个侧栏项的图标统一放进 `Width=20 + TextAlignment=Center` 的固定列，文字统一 `Margin=8,0,0,0`；分组色点也用同宽列 |
| 2 | 「新建分组」图标+文字太宽，与下方「使用说明」保持一致，并置于其上 | ✅ | 「新建分组」改用与「使用说明」相同的图标列模板，顶部间距 `8→2`；本就位于「使用说明」上方 |
| 3 | 标题栏右键「置于顶层」无效 | ✅ | 改由代码托管 `Topmost`，修复 `ShowMainWindow` 置顶技巧破坏绑定的问题 |
| 4 | 添加任务时光标与输入文本未靠左 | ✅ | `FlatTextBox` 显式 `HorizontalContentAlignment/TextAlignment=Left`，占位文字 `Margin` 对齐到 10 |

## 详细说明

### 1. 侧边栏图标 / 文字对齐
- 问题：各侧栏项图标字形宽度不一（emoji 📋📖🎨 偏宽、符号 ＋ ☰ 偏窄），且「全部任务」图标未设 `FontSize`，导致文字起始位置参差不齐。
- 方案：每个图标 `TextBlock` 统一 `FontSize=15 Width=20 TextAlignment=Center`，构成固定宽度的「图标列」；文字统一 `Margin=8,0,0,0`。
  这样无论字形宽窄，文字都从同一 x 起始（左内边距 8 + 图标列 20 + 间距 8）。
- 分组列表项：色点 `Ellipse` 列宽由 `Auto` 改为 `20` 并 `HorizontalAlignment=Center`，分组名加 `Margin=8,0,0,0`，与上方导航项文字对齐。
- 涉及：`Views/MainWindow.xaml`（全部任务 / 分组项 / 新建分组 / 使用说明 / 修改主题 / 收起侧边栏）。

### 2.「新建分组」与「使用说明」一致

### 1. 自动贴边功能修复

#### 问题分析
- **左右贴边失效**：原代码已有左右贴边检测逻辑，但触发区域判断有问题
- **上边触发区域过窄**：隐藏态下仅在窗口横向范围内才触发，应该整个屏幕上边缘都能触发
- **误隐藏问题**：显示态下缓冲区仅 6px，鼠标稍微移动就触发隐藏
- **Windows手势冲突**：拖到边缘触发系统的 Aero Snap 自动最大化/分屏

#### 修复方案
1. **触发区域优化**（`MainWindow.xaml.cs#EdgeProbeTick`）：
   - 上边：去掉横向范围限制，鼠标到达屏幕上边缘任意位置即触发
   - 左边：去掉纵向范围限制，鼠标到达屏幕左边缘任意位置即触发
   - 右边：去掉纵向范围限制，鼠标到达屏幕右边缘任意位置即触发
   - 触发条件从 `pos.Y >= wa.Top - 1 && pos.Y <= wa.Top + 2 && pos.X >= Left && pos.X <= Left + Width` 改为 `pos.Y >= wa.Top - 1 && pos.Y <= wa.Top + VisibleStripPx + 2`

2. **缓冲区增大**：显示态下的缓冲区从 6px 增大到 20px，避免鼠标在窗口内轻微移动就触发隐藏

3. **禁用 Aero Snap**（`NativeMethods.cs` + `MainWindow.xaml.cs#OnLoaded`）：
   - 新增 `DisableAeroSnap` / `EnableAeroSnap` 方法，通过移除 `WS_MAXIMIZEBOX` 窗口样式禁用系统贴边手势

### 3. 父子待办逻辑优化

#### 数据模型增强
- `TodoItem` 新增 `ParentId` 字段（`Guid?`），建立明确的父子关系
- 原有的 `IndentLevel` 仅用于视觉缩进，`ParentId` 用于逻辑关联

#### 父完成 → 子自动完成
- `MainViewModel#OnItemPropertyChanged`：父待办完成时调用 `CompleteChildren` 递归完成所有子待办
- `CompleteChildren` 方法：查找所有 `ParentId == parent.Id` 的子待办，设置 `IsCompleted = true`，并递归处理子待办的子待办

#### 子完成 → 只放烟花
- `MainWindow#OnTaskCompleting`：判断 `item.ParentId.HasValue`，如果是子待办则：
  - 播放烟花和音效（如果设置开启）
  - **不播放**滑出消失动画
  - 直接调用 `FinishCompletion` 移入"已完成"分组
  - 调用 `CheckParentCompletion` 检查父待办是否应自动完成

#### 全部子完成 → 父自动完成
- `MainViewModel#CheckParentCompletion`：
  - 查找父待办的所有子待办（`ParentId == parent.Id`）
  - 如果所有子待办都已完成（`siblings.All(s => s.IsCompleted)`），自动完成父待办
  - 递归检查父待办的父待办，支持多层嵌套

#### 动画适配
- 父待办完成：播放完整的"滑出 + 淡出 + 行收起"动画 + 烟花特效
- 子待办完成：仅播放烟花特效，不播放消失动画（视觉上保持在列表中，等父待办完成时一起消失）

### 实现要点

#### 1. 移除语音输入

- Win+H 在不同 Windows 设置 / 窗口焦点状态下行为不稳定，按 `SendInput` 模拟仍存在”按下没反应”的场景，故按题目要求直接移除该功能：

#### 2. 父子关系联动 & 整族消失动画

- **父子关系自动推断**：以 `IndentLevel` 与同分组内的 `OrderIndex` 顺序为唯一来源，每次”任何会变动层级或顺序”的操作后立即调用 `MainViewModel.RecomputeParents()`：
  - `ChangeIndent`（Tab / Shift+Tab / 右键缩进）
  - `AddTask`（新建任务）
  - `DropTask`（拖拽排序）
  - `ImportMarkdown`（导入）
  - 启动时一次性重算，修正旧数据。
  - 算法：分组内按 `OrderIndex` 顺序遍历，维护一个”层级栈”，遇到一项时弹出 ≥ 自己层级的项，栈顶即为父；`IndentLevel=0` 时 `ParentId=null`。
- **完成行为**（`MainViewModel.OnItemPropertyChanged`）：
  - 子待办勾选：若其父存在且未完成，则**只打钩、不移动到”已完成”分组**，并调用 `CheckParentCompletion` 检查父待办；
  - 父或独立待办勾选：调用 `MarkDescendantsCompletedSilently` 静默把所有后代设为完成（`_completingFamily` 标志抑制重入），随后抛出 `TaskCompleting` 事件给视图层；
  - 当父待办的所有直接子待办均已完成 → 自动把父待办勾上（递归触发上面的”父分支”，从而触发整族消失动画）。
- **整族消失动画**（`MainWindow.xaml.cs#OnTaskCompleting`）：
  - 收集 `Vm.CollectFamily(parent)`（父 + 所有递归子）；
  - 对每个仍在当前视图中的容器同时播放原有 1.1s 的”向左滑出 + 淡出 + 行高收起”动画，并叠加烟花特效；
  - 全部动画结束后回调 `Vm.FinishFamilyCompletion(parent)`，把整族一次性搬到”已完成”分组并刷新列表。
- 关键文件：`Models/TodoItem.cs`（`ParentId / IsCollapsed / HasChildren`）、`ViewModels/MainViewModel.cs`（`RecomputeParents / MarkDescendantsCompletedSilently / CollectFamily / FinishFamilyCompletion`）、`Views/MainWindow.xaml.cs#OnTaskCompleting`。

### 1. 拖动后立即刷新折叠箭头 + 美化 + 添加子待办展开父

- **修复**:`DropTask` 纵向分支末尾显式补 `RefreshItems();`。拖拽进行中该刷新经 `_pendingRefresh` 挂起,`IsDragging=false`(拖拽结束)时立即补刷,符合"拖动后立刻刷新"。
- **添加子待办展开父**:`AddTask` 创建子待办后,对父项 `parent.IsCollapsed = false` 再 `RefreshItems`,新子项立即可见。
- **箭头美化**:折叠按钮改用自定义 `ControlTemplate`——圆角(5px)、悬停 `CardHoverBg` 高亮背景、20×20 命中区;字形换更柔和的 ▾(展开)/▸(折叠),折叠态用 `Accent` 前景、展开态 `SecondaryText`。

# 第二十七轮优化（单实例跨版本接管修复 + 仅驻留通知栏）日期：2026-06-02（v1.1.3 重发布）

## 本轮原始提示词
> 我之前让你写了个待办项目，帮我完成以下需求，然后记录到优化记录.md（该文档在 MinimalTodoApp 文件夹下面已存在，记录下原来的提示词），并且输出无依赖的 exe，然后将所有改动推送到 GitHub（注意国际化），并且更新这次改动新增到 commit message，本次修改版本号保持为 v1.1.3，重新 release
> 1、应用启动时显示旧版本在运行中，点击退出启动新版本后旧版本没有退出，另外这个退出机制对不同版本号是否兼容，请都测试一下

## 总览

| # | 需求 | 状态 | 关键实现 |
|---|------|------|----------|
| 1 | 「退出旧版本接管」点了却没退出 + 跨版本号兼容 | ✅ | 根因=旧逻辑按**完整进程名精确匹配**杀进程，而发布 exe 文件名带版本号（`MinimalTodoApp-vX.Y.Z-win-x64`）→ 进程名各不相同 → 永远匹配不到旧进程。改为三层接管：命名事件**优雅信号** + **PID 文件**精确定位 + **进程名前缀**兜底，全部与版本号/文件名无关 |
| 2 | 运行时不进任务栏，只在通知栏 | ✅ | 主窗口 `ShowInTaskbar="False"`；最小化按钮改为隐藏到托盘（无任务栏可去） |

## 详细说明

### 1. 单实例「退出旧版本并接管」跨版本号修复
- **根因**：`App.EnsureSingleInstance` 用固定 GUID 互斥体检测旧实例（这步本就与版本无关，正确），但确认接管后调用的 `KillOtherInstances` 用的是
  `Process.GetProcessesByName(current.ProcessName)` —— 按**当前进程名精确匹配**。而 `release.ps1` 发布的资产名是 `MinimalTodoApp-v1.1.3-win-x64.exe`（版本号写进了文件名 → 进程名），于是旧版 `MinimalTodoApp-v1.1.2-win-x64` 与新版 `MinimalTodoApp-v1.1.3-win-x64` **进程名不同**，精确匹配永远落空：旧进程杀不掉 → 新实例 `WaitOne` 5 秒超时 → 新实例自己 `Shutdown` 退出。表现就是「点了退出旧版本，旧版本没退、新版本也没起来」。**这套机制此前正是对不同版本号不兼容**。
- **修复**（`App.xaml.cs` 重写，全部命名内核对象用**固定 GUID**，不含版本号/文件名）：
  1. **优雅信号**：新增命名事件 `MinimalTodoApp_ExitSignal_{GUID}`。活动实例在 `BecomeActiveInstance` 用 `ThreadPool.RegisterWaitForSingleObject` 监听它；新实例接管时 `SignalOtherInstancesToExit` 打开同名事件并 `Set()`，旧实例收到后经 `MainWindow.ForceExit()` 保存数据→释放托盘→干净退出。
  2. **PID 文件**：活动实例把自身 PID 写到 `%AppData%\MinimalTodoApp\instance.pid`；新实例据此 `Process.GetProcessById` 精确结束旧实例（与 exe 文件名完全无关）。为防 PID 复用误杀，杀前校验目标进程名仍以 `MinimalTodoApp` 前缀开头。
  3. **进程名前缀兜底**：枚举所有进程，凡进程名以 `MinimalTodoApp` 开头者（含 `MinimalTodoApp-v1.1.2-win-x64` 这类**已发布的旧版本**，它们既不监听事件也不写 PID）一律结束。这一层是对「线上已存在的旧版本」的关键兼容兜底。
  4. 三层依次尝试，每步后 `WaitOne` 接管互斥体；旧实例退出后互斥体 abandoned，捕获 `AbandonedMutexException` 即视为接管成功。
- **MainWindow**：抽出 `public void ForceExit()`（允许真正关闭→回写侧栏宽度→释放托盘→`Shutdown`），托盘「退出」菜单与「被新版本优雅接管」共用此入口。

## 详细说明

### 1. 检查与提示（`Infrastructure/UpdateService.cs`）
- `CheckAsync()`：`HttpClient`（带 `User-Agent`，GitHub 强制）GET `releases/latest`，用 `System.Text.Json` 取 `tag_name`/`body`/`assets`。
  tag 去前导 `v` 解析为 `Version`，仅按 主.次.修订 三段比较；选名字以 `win-x64.exe` 结尾的资产。**仅当 latest > current 才返回**，否则/异常返回 `null`（静默，绝不抛）。
- 当前版本取自 `Assembly.GetEntryAssembly().GetName().Version`；当前 exe 路径取 `Environment.ProcessPath`。
- `UpdateDialog`：展示「发现新版本 X（当前 Y）」+ 可滚动的 Release 说明（`body`），三按钮「立即更新 / 忽略 / 此版本不再提示」。

### 2. 下载与重启（关键：文件名带版本号）
- `DownloadAsync()`：`ResponseHeadersRead` 流式下载，按真实资产名（如 `MinimalTodoApp-v1.1.4-win-x64.exe`）落到**当前 exe 同目录**（不可写则退到 `%LOCALAPPDATA%\MinimalTodoApp`），`IProgress<double>` 回报进度。
- `LaunchUpdaterAndExit()`：在 `%TEMP%` 生成临时 `.cmd`——**等待旧版 PID 退出 → `start` 新版 exe 并附 `--updated-from "<旧exe>"` → 自删脚本**；随后 `MainWindow.ForceExit()` 干净退出（释放托盘+互斥体）。脚本用 `tasklist /FI "PID eq X" | find` 判断旧进程是否还在。
- 因旧版已退出、互斥体已释放，新版被脚本拉起时**不会触发单实例「是否结束旧实例」弹窗**。
- 新版启动（`MainWindow.InitAutoUpdate`）解析 `--updated-from`：后台线程把旧版 exe **移入回收站**（`NativeMethods.MoveToRecycleBin` → `SHFileOperation` + `FOF_ALLOWUNDO`，带短重试等文件解锁）。
- **开机自启动指向新版**：无需新增代码——`MainViewModel.EnsureFirstRunStartup()` 每次启动已调 `StartupManager.SyncRegisteredPath()`，发现 Run 注册表项路径与当前 exe 不一致即改写为新版路径。

### 3. 设置项（`Views/SettingsDialog`）
- 「常规」组新增：**自动检查更新** 开关（绑定 `MainViewModel.AutoUpdateEnabled`，持久化）；**检查更新** 按钮（手动，`CheckUpdate_Click`）。
- 手动检查：有新版弹 `UpdateDialog`；已是最新→状态文案「已是最新版本」；失败→「检查更新失败，请稍后再试」。手动检查**无视**「此版本不再提示」。

### 4. 自动检查时机（`Views/MainWindow.xaml.cs`）
- `InitAutoUpdate()`：① 处理 `--updated-from` 回收旧版；② 每小时 `DispatcherTimer` 检查；③ 启动后延迟 4 秒检查一次（不打断启动）。
- `CheckForUpdateSilentlyAsync()`：发现新版且未被 `IgnoredUpdateVersion` 跳过时弹 `UpdateDialog`；`_updateChecking` 防重入；全程 try/catch 静默。
- 关闭「自动检查更新」后，定时与启动检查都直接跳过（仅手动可触发）。

### 5. 旧版 release 一并升级（保证旧版也能自动更新）
- **前提（物理限制）**：已被用户下载到本地、不含更新代码的旧副本无法被追回加功能。下述只让**今后从任一 Release 页面重新下载**的人获得自动更新能力。
- 用当前含功能的同一份源码，分别 `dotnet publish -p:Version=<该页版本>` 打出 5 个自包含单文件 exe，重命名为各页资产名，**替换** v1.0.0~v1.1.3 各 Release 的资产（GitHub API 删旧资产→传新资产，不动 tag）。

### 补充修复：点更新后新版没启动（最终方案：隐藏 PowerShell 拉起）
- **现象**：用户实测——下载完、旧版退出后，新版应用没有被拉起。
- **定位历程**：
  1. 第一版用 cmd 脚本 + `start "" "newexe"`。实测 `start` 在 `CreateNoWindow`（无控制台）下失败（errorlevel 1），新版起不来；`timeout /t` 同理需控制台输入会报错。

  - 轮询 `Get-Process -Id <旧版PID>` 等其退出（最多 ~60 秒兜底）；
  - 用 `Start-Process`（不带 `-Wait`）拉起新版 exe 并附 `--updated-from "<旧exe>"` —— 立即返回、新进程独立运行，不阻塞、不残留；
  - 脚本 `Remove-Item` 自删。路径用 PowerShell 单引号转义安全嵌入。
- **验证**：harness 引用**真实重编译后的 DLL** 调真实 `LaunchUpdaterAndExit`，目标为**真实自包含单文件 v1.1.3**，干净单实例环境下：新版进程起来且**可见窗口 title=`Todo`** ✓、无残留 updater PowerShell 进程 ✓、临时 ps1 已自删 ✓。

## 各项详细说明

### 1. 体积（受限说明）
- `.NET 8 + WPF` 自包含单文件已开启 `EnableCompressionInSingleFile`、`SatelliteResourceLanguages=en`、`DebugType=none`。
- WPF 应用启用 `PublishTrimmed` 会被 SDK 直接报错/破坏 XAML 加载，**不可用**；改框架依赖会破坏"无依赖"硬约束。故按用户选择维持极保守优化，体积约 ~60MB。

### 2. 子待办刷新 / 显示一致（`ViewModels/MainViewModel.cs`）
- `OnItemPropertyChanged` 中"活子待办"（父未完成）勾选分支原来只 `RefreshGroupCounts()+SaveData()`，新增 `RefreshItems()`，使父行"0/2 → 1/2"即时刷新。
- `RefreshItems()` 中"所有待办"过滤改为 `query.Where(i => !RootOf(i).IsCompleted)`：根未完成的家族整体显示（含其下已完成子待办），与普通分组完全一致；根完成的家族已迁入"已完成"分组自然不再出现。

### 3. 右键菜单白条（`Themes/Controls.xaml`）
- 系统默认 `MenuItem` 模板含图标/勾选 gutter 列（即左侧白色矩形）。改用自定义 `ControlTemplate`：仅"标题列 + 子菜单箭头列"，悬停 `CardHoverBg`、禁用 `MutedText`、圆角 6；`ContextMenu` 亦自定义模板（圆角 8 + 阴影）。"移动到分组"子菜单通过 `Role=SubmenuHeader` 触发箭头与 `Popup` 子菜单；其 `ItemContainerStyle` 用 `BasedOn` 继承新模板，避免子项回退系统样式重现白条。

### 4. 分组层级缩进（`Models/TodoGroup.cs` + `Views/MainWindow.xaml`）
- 新增只读 `Thickness IndentMargin`：所有待办/已完成 → 0；普通分组 → `16,0,0,0`。分组行 `Grid` 绑定 `Margin`；折叠侧栏（窄条）时 DataTrigger 归零，避免图标被挤偏裁切。

### 5. 精简右键菜单（`Views/MainWindow.xaml`）
- 移除四项菜单：清除截止日期、折叠/展开子待办、切换周期提醒、标记完成/未完成。保留：编辑、置顶、增/减缩进、移动到分组、删除。底层 `RelayCommand` 保留（折叠仍可用任务行右侧箭头按钮）。

### 6. 拖到分组移动（`ViewModels/MainViewModel.cs`）
- `DragOver` 新增：`Data is TodoItem && TargetItem is TodoGroup`（且目标为可归属普通分组）→ 接受 Move + 高亮 adorner。`Drop` 新增同分支 → 调用既有 `MoveTaskToGroup`（已处理已完成→取消完成、追加排序、刷新、保存）。

### 7. 编辑弹窗拖动（`Views/TaskEditDialog.xaml(.cs)`）
- 根 `Border` 加 `MouseLeftButtonDown="Root_MouseLeftButtonDown"`，后端 `DragMove()`（try/catch 容错；点输入框由控件自身处理不触发拖动）。

### 8. 圆角统一（`Themes/Controls.xaml` 等）
- 复核全部窗口与自定义 Popup：均已 `AllowsTransparency=True` + 圆角 `Border`。系统弹层（`ToolTip`/`ComboBox` 下拉）原为直角，新增圆角主题化模板（`ToolTip` 圆角 6、`ComboBox` 下拉圆角 8 + 阴影 + 圆角项）。

### 9. DPI 清晰度（`app.manifest` + `MinimalTodoApp.csproj` + `Themes/Controls.xaml`）
- 新增 `app.manifest` 声明 **Per-Monitor V2** DPI 感知，避免系统对窗口位图拉伸导致 1080p / 非整数缩放下文字发虚；csproj 加 `<ApplicationManifest>`。

## 补丁：检查更新误报"已是最新" + 放宽检查频率（v1.1.4 / v1.1.3 资产同步）

> 日期：2026-06-04。用户反馈：v1.1.3 点"检查更新"显示已是最新，但 v1.1.4 已发布。

**根因**：
1. App 用**匿名**方式（无令牌）调 GitHub `releases/latest`，受 **60 次/小时/IP** 限流；额度耗尽时返回 **HTTP 403**，根本没拿到最新版本信息。
2. `UpdateService.CheckAsync` 把"网络/HTTP 失败"与"确实已是最新"都 `return null`，设置里手动检查拿到 null 一律显示"已是最新"，于是 403 被误报为"已是最新"。

**修复**：
- `Infrastructure/UpdateService.cs`：`CheckAsync` 改为——`resp.EnsureSuccessStatusCode()` 让非 2xx（含 403 限流）**抛异常**，网络/解析异常一并向上抛；`null` 仅表示"已是最新 / 无合适资产"。后台静默检查（`MainWindow.CheckForUpdateSilentlyAsync`）与手动检查（`SettingsDialog.CheckUpdate_Click`）本就各有 try/catch：后台静默忽略，手动则显示"检查失败，请稍后重试"（`S.Update.CheckFailed`），不再把失败说成已是最新。
- `Views/MainWindow.xaml.cs`：定时检查频率由 **每 1 小时**放宽到 **每 12 小时**（启动后延迟数秒仍检查一次），降低撞限流概率。

**发布**：以修复后的构建**替换 v1.1.4 与 v1.1.3 两个 Release 的 win-x64.exe 资产**（v1.1.3 资产从 v1.1.3 tag 源码 + 同一补丁忠实重建，版本仍标 1.1.3；v1.1.4 资产版本 1.1.4）。这样无论用户下载哪个版本，更新检查逻辑均已修正、频率为 12 小时。
- 注意：已安装在用户机器上的旧 v1.1.3（带 bug）无法被资产替换改动，仍需手动下载一次修复版或等限流重置后手动检查；今后版本不再有此误报。
